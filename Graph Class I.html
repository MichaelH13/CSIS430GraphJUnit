Graph Class I

In this course we'll be working with graphs. So let's build a simple templated graph class with some basic graph functionality that we can expand upon in later assignments. We'll implement a weighted graph that can be directed or undirected: 

public class Graph<T> {
   public Graph(boolean isDirected) {}
   public List<T> getVertices() {}
   public int getEdgeWeight(T source, T destination) {}
   public boolean edgeExists(T source, T destination) {}
   public void addVertex(T vertex) throws IllegalArgumentException {}
   public void removeVertex(T vertex) throws NoSuchElementException {}
   public void addEdge(T source, T destination, int weight) 
         throws IllegalArgumentException, NoSuchElementException {}
   public void removeEdge(T source, T destination) throws NoSuchElementException {}
   public String toString() {}
}

Notes:

  • If the graph is undirected then adding an edge between V1 and V2 means there is also an edge between V2 and V1. Similarly, removing either edge conceptually removes the other.
  • Some methods should throw appropriate exceptions as shown above for reasons that should be obvious. A less obvious error is that it's not legal to add an edge between a vertex and itself. Otherwise, some methods don't need to throw exceptions as their return value is sufficient (e.g., null or -1). Overwritting an existing edge (i.e., updating weight) is fine as well.
  • getEdgeWeight should return -1 if no such edge is present as weighted graphs often (and in our case) only allow non-negative edge weights. In addition, though we won't add edges where source and destination are the same, if asked for the weight of such an edge we'll respond with 0.
  • The class, as shown, says nothing about how the graph is to be implementation. You might choose an Adjacency-Matrix or Adjacency-List implementation. Yet another alternative are nested HashMap which is somewhat like an Adjacency-List implementation with slightly higher space complexity but slightly better time complexity; two benefits are that we can use non-numeric values for vertices and the underlying data structure does most of the work for us when implementing methods.
  • It may seem odd to have getVertices() return a sequence instead of a set. The reason for that may become clear later; otherwise a set would have been quite reasonable.
  • toString() is left as an exercise for the student; it should return a reasonable human-readable representation of the graph.

This would be a good opportunity to practice test-driven development by writing a JUnit test-harness first and then implementing your class.

Finally, this is relatively simple and straightforward; most methods (not including exception-handling) should be very short. Do not over-design but understand and fully meet the specifications. If you have questions about the specifications, ask!
